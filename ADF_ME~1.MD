## 1. 전체 구성

메타데이터 기반(metadata-driven) 오케스트레이션 구조입니다. 수집 대상 정보를 메타 테이블에서 관리하고, 마스터 파이프라인이 이를 읽어 수집 파이프라인을 반복 실행한 뒤, Databricks Bronze 적재까지 연계·확인합니다.

**메타 테이블 (개요)**

| 테이블 | 역할 | 주요 항목 |
|---|---|---|
| 수집대상마스터 | 수집 대상의 영속 정의 | 대상 정보, 추출쿼리 기준, 랜딩경로, `pending` 플래그 |
| 마스터파이프라인 수행상태 | 마스터 실행 단위 상태 | 파이프라인명, run-id, 트리거명, 파라메터 payload, 수집 대상 수, 성공/실패 개수, 시작·종료일시, 상태 |
| 수집파이프라인 수행이력 | 수집 대상별 실행 이력 | 마스터 run-id, 대상 정보, 추출쿼리, 랜딩경로, 파일명, 추출건수/저장건수, 시작·종료일시, 상태 |

**파이프라인 (개요)**

| 파이프라인 | 용도 |
|---|---|
| 마스터 파이프라인 | 정규 증분(incr) 배치 수집 오케스트레이션 |
| 수집(child) 파이프라인 | 대상 1건 실제 추출·랜딩 (마스터 ForEach 내부에서 호출) |
| pending 업데이트 파이프라인 | 초기(full) 긴급/테스트 재수집 대상 지정 |

## 2. 마스터 파이프라인 (정규 증분 배치)

정규 스케줄로 수행되며, 필요 시 파라메터로 특정 대상만 수동·긴급 재수행할 수 있습니다.

### 파라메터 (선택 입력)

- **용도**: 특정 테이블 수집 테스트, 또는 긴급 수작업 재수행
- **incr 수집 대상 정보** (JSON):

```json
[
  { "target_id": 101, "condition": "2026-07-01 00:00:00" },
  { "target_id": 300 }
]
```

- `condition` 값이 있으면 해당 대상의 **증분 조건(watermark)을 override** 합니다.

### 수행 흐름

| 단계 | 활동 | 처리 내용 |
|---|---|---|
| 0 | 변수 설정 | 파라메터 유무에 따라 1단계에서 수행할 스크립트를 분기 |
| 1 | Script (SP) | **마스터파이프라인 수행상태** insert (파이프라인명, run-id, 트리거명, 파라메터 payload, 수집 대상 수, 시작일시, 상태 등)<br>**수집파이프라인 수행이력** insert (마스터 run-id, 대상 정보[수집대상마스터에서 조회], 추출쿼리, 랜딩경로 등) |
| 2 | Lookup | **수집파이프라인 수행이력** 조회 — full/incr 타입별 쿼리 |
| 3 | ForEach | 대상별 반복 (하위 3.1~3.3) |
| 3.1 | Script (SP) | 수집이력 update — child run-id, 파일명, 시작일시 |
| 3.2 | Execute Pipeline | **수집(child) 파이프라인 수행** — 실패 시 최대 2회 재시도 |
| 3.3 | Script (SP) | 수집이력 update — 종료일시, 추출건수, 저장건수, 상태 |
| 4 | Script (SP) | **마스터파이프라인 수행상태** update — 성공/실패 개수, 종료일시, 상태 |
| 5 | Web Activity | Databricks 노트북 실행(REST API) — 배치 수집 완료 알림 (마스터 run-id 전달) |
| 6 | Until + Wait + Web Activity | Databricks Bronze 수집 완료 확인 (REST API 폴링) |

> **설계 원칙**
> - **step 4·5 수행 보장**: 3.2가 재시도 후에도 실패하더라도 ForEach가 실패로 종료되지 않도록 처리하여(실패는 3.3 이력에 기록), 마스터 상태 update(4)와 Databricks 알림(5)이 반드시 수행되도록 함.
> - **step 6 폴링**: Until에 timeout/최대 반복을 두어 무한 대기를 방지하고, 5단계에서 받은 Databricks run-id로 완료 상태를 확인.

## 3. pending 업데이트 파이프라인 (초기 수집 긴급/테스트)

초기(full) 수집 테스트 대상, 또는 긴급하게 다시 초기 수집해야 하는 대상을 지정하는 파이프라인입니다. 마스터 파이프라인과 분리하여 **수집대상마스터의 `pending` 플래그만 갱신**합니다.

### 파라메터

- **full 수집 대상 정보** (JSON):

```json
[
  { "target_id": 101 },
  { "target_id": 102 }
]
```

### 수행 흐름

| 단계 | 활동 | 처리 내용 |
|---|---|---|
| 1 | Script (SP) | **수집대상마스터** update — 전달된 대상의 `pending = 'Y'` 설정 |

> 이후 마스터/초기 수집 파이프라인이 `pending = 'Y'` 대상을 편입하여 수집하며, 대상 편입 시점에 `pending`을 리셋합니다.

---

## 4. 협의 필요 사항

- **incr override watermark 정책**: 수동 재수행에서 `condition`으로 증분 조건을 override했을 때, 성공 후 정규 watermark(`last_run_dt`)를 전진시킬지 여부는 **고객과 협의 필요**. (긴급 재수집의 경우 전진시키지 않으면 정규 스케줄 증분 구간의 누락을 방지할 수 있음)
