# 메타 테이블 기반 증분 수집 — 증분 쿼리 조건 생성 방식 검토서

> 목적: 메타 테이블 기반 증분 수집 시 **증분 조건(날짜 상/하한) 생성 방식**을 확정하기 위한 고객 컨펌용 검토 자료
> 비교 대상: **A. ADF 표현식 계산 방식** vs **B. 소스 DB 함수 직접 수행 방식**
> 작성일: 2026-07-23

---

## 0. 요약 (Executive Summary)

이기종 소스 DB(DB2/MS-SQL/Azure COSMOS/PG 등)에서 증분 컬럼 기준으로 데이터를 추출할 때, 증분 조건의 날짜 값을 **어디서·어떤 형태로 생성**할지 다음 2가지 방안을 비교한다.

**A. ADF 표현식 계산 방식**
증분 날짜 range의 실제 값을 **모두 ADF 함수로 계산**한다. 계산 결과를 **파라미터로 메타 DB에 전달** → 메타 테이블 컬럼(`incr_start_dt`/`incr_end_dt`)에 저장하고, 그 **상수 값**으로 추출 쿼리를 생성한다.
→ 추출 쿼리 예: `select * from schema.tab where ipdtm >= '2026-07-23 00:00:00' and ipdtm < '2026-07-24 00:00:00'`

**B. 소스 DB 함수 직접 수행 방식**
메타 PG 쿼리가 **소스 DB 네이티브 함수를 인라인으로 포함한** 추출 쿼리를 생성한다. 날짜 계산은 소스 DB가 실행 시점에 수행한다.
→ 추출 쿼리 예: `select * from schema.tab where ipdtm >= current date - 1 days and ipdtm < current date`

**핵심 판단축**

- **추적/재현성** — A는 실행 구간이 상수로 메타에 남아 감사·백필에 유리. B는 함수라 실행 시점에 재계산되어 사후 구간 확인이 어렵다.
- **이기종 대응** — A는 표준 문자열 상수라 소스 DB 종류 무관. B는 DB2/Oracle 등 dialect별 함수 문법을 개별 관리해야 한다.
- **메타 쿼리 단순성** — B가 가장 단순(함수 문자열 concat). A는 파라미터 조립 + 타입 캐스트가 필요하나, 메타에서 날짜 연산은 하지 않아 여전히 단순하다.
- **타임존** — A는 ADF가 기본 UTC라 대상 TZ 변환을 명시해야 함. B는 소스 시스템 시간을 그대로 써서 자연 정확.

> 참고: 두 방안을 **빈도별로 혼용**할 수도 있다(예: 고빈도 minutely/hourly → A, 저빈도 daily+ → B). 자세한 선택 기준은 2절 결론 참조.

## 1. 배경 및 아키텍처

- 소스 DB(이기종: DB2/MS-SQL/Azure COSMOS/PG 등)에서 증분 컬럼(`incr_column_nm`, 예: `ipdtm`) 기준으로 데이터를 추출한다.
- 파이프라인 실행 시 메타 DB(PostgreSQL)의 `INSERT ... SELECT`로 `ctl_ingest_pipeline_run` 레코드를 생성하며, 이 순간 `extract_query`가 확정된다.
- ADF는 오케스트레이션(트리거·파라미터 전달)을 담당한다.
- 쟁점: 증분 조건의 날짜 값을 **A(ADF 계산 후 메타 상수 저장)** 로 만들지, **B(메타가 소스 함수 인라인 쿼리 생성)** 로 만들지.

## 2. 두 방안 비교

| 비교 항목 | A. ADF 표현식 계산 방식 | B. 소스 DB 함수 직접 수행 방식 |
|---|---|---|
| **날짜 계산 위치** | ADF 함수(트리거 시각 기준) | 소스 DB(추출 실행 시점) |
| **메타 DB 역할** | ADF가 넘긴 start/end 파라미터를 컬럼에 저장 + 추출 쿼리 조립 | 소스 함수를 포함한 추출 쿼리 문자열 생성(concat) |
| **추출 쿼리 형태** | 상수 비교 `ipdtm >= '2026-07-23 00:00:00'` | 소스 함수 인라인 `ipdtm >= current date - 1 days` |
| **메타 쿼리 복잡도** | 낮음~중간 (파라미터 조립 + 타입 캐스트, 날짜 연산 없음) | 최소 (함수 문자열 concat) |
| **소스 DB 부하** | 최소 — 상수 비교로 인덱스 스캔 최적화 | 주의 — 인라인 함수가 인덱스 미사용 유발 가능 |
| **이기종 소스 DB 대응** | 우수 — 표준 문자열 상수, dialect 무관 | 취약 — DB2/Oracle 등 함수 문법 개별 구현·관리 |
| **타임존** | ADF 기본 UTC → 대상 TZ 변환 명시 필요 | 소스 시스템 시간 그대로 → 자연 정확 |
| **이력 추적 / 재현** | 최상 — 실행 구간이 상수로 메타에 보존, 복사해 즉시 재현 | 미흡 — 함수라 나중에 실행하면 재계산, 당시 구간 확인 불가 |
| **백필 / 재처리** | 용이 — ADF 파라미터를 과거 값으로 넘기면 상수 교체 | 가능하나 Override 구현 전제(함수가 고정이라) |
| **자정/경계 처리** | ADF trigger 시각 기준(트리거 스케줄로 통제) | 소스 실행 시각 기준(실행 지연·자정 주의) |
| **운영자 체감** | ADF·메타 2곳을 봄. 값이 상수라 디버깅은 명확 | 메타 쿼리는 짧으나, 실제 구간은 실행 시점에 결정 |

### 결론 (선택 기준)

- **추적성·백필·이기종 대응이 중요하면 → A.** 실행 구간이 상수로 남아 "이 시각에 무엇을 조회했는가"가 명확하고, 소스 DB 종류가 늘어도 문법 오류가 없다.
- **메타 쿼리 최소화·소스 시간 정확성이 중요하면 → B.** 메타는 함수 문자열만 붙이면 되고, 소스가 자기 시간·문법으로 계산해 TZ·포맷 고민이 없다. 단 추적성·이기종 유지보수는 약하다.
- **혼용(권장 옵션):** 고빈도(minutely/hourly)는 ADF 계산이 가볍고 소스 왕복이 없어 **A**, 저빈도(daily+)는 소스 시간 정확성이 필요하고 CLYM 같은 회계 달력을 소스가 처리하므로 **B** — 이렇게 나누는 것도 실무적으로 합리적이다.

## 3. 상한선(`<`) 조건의 필요성 (A·B 공통)

두 방안 모두 하한선만(`ipdtm >= '시작'`) 두지 말고, 반드시 **`>= 시작 AND < 종료`** 로 상한선을 포함해야 한다.

### 3-1. 데이터 정합성 (누락·중복 방지)

배치가 수행되는 수초~수분 동안에도 소스 DB에는 신규 데이터가 계속 INSERT된다. 상한선이 없으면 실행 도중 유입된 데이터까지 포함되어 다음 배치에서 **중복 수집**되거나 경계 차이로 **누락**될 수 있다.

### 3-2. 배치 주기 간 경계 완벽 분리

`>=`(이상) + `<`(미만) 조합으로 시간 슬라이스가 겹침·틈 없이 연속 분할된다.

| 배치 | 조회 범위 |
|---|---|
| 12시 배치 | `11:00:00 <= ipdtm < 12:00:00` |
| 13시 배치 | `12:00:00 <= ipdtm < 13:00:00` |

### 3-3. 백필/재처리 안전성

과거 특정 구간만 재적재할 때 상한선이 없으면 시작점부터 현재까지 전부 재수집되어 타겟이 오염된다.

> 참고: B 예시로 제시된 `where ipdtm >= current date - 1 days`는 하한선만 있으므로, 실제 적용 시 `and ipdtm < current date` 상한선을 반드시 추가한다.

### 3-4. 지연 도착(late-arriving) 데이터와 lookback 정책

`>= 시작 AND < 종료`는 **증분 컬럼 값(`ipdtm`) 기준**으로 자르는데, 이 값과 **실제 DB에 commit되어 조회 가능해지는 시각**이 어긋날 수 있다. 이 틈에서 데이터가 유실된다.

- 트랜잭션이 `ipdtm = 11:59:50`으로 시작됐으나 commit은 12:00:03에 발생
- 12:00 배치가 `< 12:00:00`으로 조회하는 순간엔 아직 안 보여 **이번에 누락**
- 다음 13시 배치는 `>= 12:00:00`부터 보므로 이 row(`11:59:50`)는 **범위 밖 → 영구 유실**

복제 지연(replication lag), 긴 트랜잭션, 뒤늦은 보정 INSERT가 모두 같은 문제를 만든다. 이를 막는 안전장치가 **lookback 정책**이며, 두 가지 방식이 있다.

**① 상한선 뒤로 물리기 (지연 버퍼)** — 종료 경계를 현재가 아니라 `현재 - N분`으로 잡아, N분 내 commit될 데이터를 다음 배치로 넘긴다.

```
-- 예: 5분 지연 버퍼 (12:00 실행이지만 상한은 11:55)
where ipdtm >= '11:00:00' and ipdtm < '11:55:00'
```

유실은 줄지만 데이터가 그만큼 **늦게 수집**된다(지연 ↔ 유실 트레이드오프).

**② 하한선 겹치기 (lookback window)** — 다음 배치의 시작을 직전 구간보다 앞으로 당겨, 겹치는 구간을 다시 훑는다.

```
-- 직전 종료 12:00 인데 10분 겹쳐서 시작
where ipdtm >= '11:50:00' and ipdtm < '13:00:00'
```

겹친 구간에서 늦게 도착한 row를 잡는다. 대신 **중복이 발생**하므로 타겟에서 **PK 기준 UPSERT/dedup이 전제**되어야 한다.

| 방식 | 효과 | 비용 |
|---|---|---|
| ① 상한선 뒤로 물리기 | 유실 감소 | 수집 지연 증가 |
| ② 하한선 겹치기 | 유실 감소 | 중복 발생 → 타겟 dedup/UPSERT 필요 |

실무에선 보통 **②(겹침) + 타겟 UPSERT** 조합을 쓰고, append-only로 중복을 못 받는 구조면 **①(지연 버퍼)**을 쓴다.

⚠ 확인 필요:

- **증분 컬럼의 의미** — `ipdtm`이 "행 생성/commit 시각"인지 "업무상 발생 시각"인지에 따라 지연 위험이 다르다. commit 시각에 가까울수록 lookback 필요성이 낮다.
- **타겟 dedup 가능 여부** — 겹침 방식(②)은 PK/머지 키가 있어야 안전. 없으면 지연 버퍼(①)로.
- **N(물리는 폭) 산정** — 소스의 최대 지연(복제 랙, 트랜잭션 길이)을 관측해 정한다. 과하면 지연·중복만 늘어난다.

## 4. 방안 A 상세 — ADF 표현식 계산 → 메타 저장 → 추출 쿼리 생성

### 4-1. 처리 흐름

```
[ADF] 표현식으로 start_dt / end_dt 실제 값 계산 (대상 TZ 변환 포함)
        ↓ 파라미터 전달 (P_START_DT / P_END_DT)
[메타 DB] 컬럼 저장(incr_start_dt / incr_end_dt) + 추출 쿼리 조립(상수)
        ↓
[소스 DB] 상수 조건으로 추출
```

### 4-2. ADF 표현식 예시

ADF는 기본 UTC이므로 **대상(소스) TZ로 변환 후** 절단·연산한다.

```
// 일 단위 예: end = 오늘 00:00 (대상 TZ), start = end - N일
end_dt   = @formatDateTime(convertTimeZone(utcNow(),'UTC','Korea Standard Time'),'yyyy-MM-dd 00:00:00')
start_dt = @formatDateTime(addDays(convertTimeZone(utcNow(),'UTC','Korea Standard Time'), -1),'yyyy-MM-dd 00:00:00')

// 시간 단위 예: addHours(...) 사용, 'yyyy-MM-dd HH:00:00'
```

### 4-3. 메타 DB 추출 쿼리 조립 (날짜 연산 없음)

ADF가 넘긴 상수를 저장하고, 타입 캐스트로 감싸 추출 쿼리를 만든다. **메타에서 날짜 계산은 하지 않는다.**

```sql
INSERT INTO ctl_ingest_pipeline_run (
    ..., incr_column_nm, incr_column_type,
    incr_start_dt, incr_end_dt, extract_query, ...
)
SELECT
    ..., t.incr_column_nm, t.incr_column_type,
    '@{pipeline().parameters.P_START_DT}',   -- ADF 계산 상수 저장
    '@{pipeline().parameters.P_END_DT}',
    format(
        'select * from %s.%s where %s < %s and %s >= %s fetch first 1 rows only with ur ;',
        t.src_schema_nm, t.src_table_nm,
        t.incr_column_nm,
        CASE WHEN upper(t.incr_column_type) = 'DATE'
             THEN 'DATE('      || quote_literal('@{pipeline().parameters.P_END_DT}')   || ')'
             ELSE 'TIMESTAMP(' || quote_literal('@{pipeline().parameters.P_END_DT}')   || ')' END,
        t.incr_column_nm,
        CASE WHEN upper(t.incr_column_type) = 'DATE'
             THEN 'DATE('      || quote_literal('@{pipeline().parameters.P_START_DT}') || ')'
             ELSE 'TIMESTAMP(' || quote_literal('@{pipeline().parameters.P_START_DT}') || ')' END
    ),
    ...
FROM ctl_ingest_target_master t
WHERE lower(t.is_active) = 'y' AND ... ;
```

생성 예시:

```
select * from SCHEMA.TBL where ipdtm < TIMESTAMP('2026-07-24 00:00:00') and ipdtm >= TIMESTAMP('2026-07-23 00:00:00') fetch first 1 rows only with ur ;
```

### 4-4. 특징

- 추출 쿼리가 **상수**라 소스 인덱스에 유리하고, 실행 구간이 메타에 그대로 남아 **재현·감사 최상**.
- 소스 DB 종류가 달라도 표준 리터럴이라 **문법 오류 없음**.
- 타입 캐스트(`TIMESTAMP()`/`DATE()`)로 DB2 암묵 캐스팅을 제거하고, `quote_literal(%L)`로 injection·따옴표 문제를 방지(7절).

## 5. 방안 B 상세 — 소스 DB 함수 직접 수행

### 5-1. 처리 흐름

```
[메타 DB] 소스 함수를 인라인 포함한 추출 쿼리 문자열 생성(concat)
        ↓
[소스 DB] 실행 시점에 함수 계산하여 추출
```

### 5-2. 메타 DB 추출 쿼리 생성 (소스 함수 인라인)

```sql
INSERT INTO ctl_ingest_pipeline_run ( ..., extract_query, ... )
SELECT
    ...,
    'select * from ' || t.src_schema_nm || '.' || t.src_table_nm ||
    ' where ' || t.incr_column_nm || ' >= ' || <소스 함수식(하한)> ||
    ' and '   || t.incr_column_nm || ' < '  || <소스 함수식(상한)> ||
    ' fetch first 1 rows only with ur ;',
    ...
FROM ctl_ingest_target_master t
WHERE lower(t.is_active) = 'y' AND ... ;
```

### 5-3. 소스 dialect별 함수식 예시 (일 단위)

DB2:

```sql
-- 하한: current date - N일, 상한: current date
where ipdtm >= current date - 1 days and ipdtm < current date
```

Oracle:

```sql
where ipdtm >= TRUNC(SYSDATE) - 1 and ipdtm < TRUNC(SYSDATE)
```

빈도별 함수(시/분/월/년)와 CLYM(YYMM) 등은 소스 함수로 각각 표현한다. **dialect마다 함수 문법이 다르므로 소스 유형별로 관리**해야 한다.

### 5-4. 특징

- 메타 쿼리가 **가장 단순**(함수 문자열 concat). 날짜 연산·TZ·캐스트 로직이 메타에 없다.
- 소스가 **자기 시스템 시간·문법**으로 계산 → TZ 불일치·포맷 추측 없음.
- 단, 추출 쿼리에 함수가 박혀 **실행 시점에 재계산** → 나중에 같은 쿼리를 돌리면 당시 구간이 재현되지 않음(추적성 약함).
- 인라인 함수가 **인덱스 미사용**을 유발할 수 있어 대용량 테이블은 실행계획 확인 필요.
- dialect별 함수 유지보수 부담.

## 6. 백필 / 재처리 — Override (A·B 공통)

평시에는 각 방안의 기본 로직으로 동작하되, 백필/재처리 시 ADF 파라미터로 **경계값(start/end 상수)** 이 입력되면 그 값으로 Override한다. A는 이미 상수를 쓰므로 파라미터만 과거 값으로 넘기면 되고, B는 Override 시 함수식 대신 전달된 상수로 치환한다.

### 6-1. Override 정책 표준 (확정)

마스터-차일드 구조 기준. 마스터가 Override JSON을 파라미터(`P_OVERRIDE_JSON`)로 차일드에 전달한다.

- **매칭 키:** `target_id` 고정
- **override_type:** `RANGE`(시간 범위)만 사용. `WHERE_CLAUSE`(SQL 절 통짜 전달)는 injection 위험으로 **미채택**
- **scope enum (팀 표준):**

| scope | 의미 |
|---|---|
| `GLOBAL` | 모든 대상 테이블에 공통 범위 적용 |
| `TARGET` | `targets` 지정 `target_id`만 수집(문제 테이블 수동 재수집), **나머지는 수집 안 함** |
| `GLOBAL_WITH_EXCEPTION` | `global` 범위를 기본 적용하되 `targets` 지정분은 다른 범위로 우선 |
| `GLOBAL_WITH_EXCLUDE` | `global` 범위를 적용하되 `exclude` 지정 `target_id`는 **수집 제외** |

### 6-2. JSON 예시

**① 전역 (모든 대상 공통):**

```json
{ "scope": "GLOBAL", "override_type": "RANGE", "start_dt": "2026-07-01 00:00:00", "end_dt": "2026-07-05 00:00:00" }
```

**② 특정 테이블만 수집 (수동 재수집, 나머지 수집 안 함):**

```json
{
  "scope": "TARGET",
  "targets": [
    { "target_id": "T001", "override_type": "RANGE", "start_dt": "2026-07-01 00:00:00", "end_dt": "2026-07-05 00:00:00" },
    { "target_id": "T002", "override_type": "RANGE", "start_dt": "2026-06-01 00:00:00", "end_dt": "2026-06-10 00:00:00" }
  ]
}
```

**③ 전역 + 일부 예외 (단 T002만 다른 범위):**

```json
{
  "scope": "GLOBAL_WITH_EXCEPTION",
  "global": { "override_type": "RANGE", "start_dt": "2026-07-01 00:00:00", "end_dt": "2026-07-05 00:00:00" },
  "targets": [ { "target_id": "T002", "override_type": "RANGE", "start_dt": "2026-06-01 00:00:00", "end_dt": "2026-06-10 00:00:00" } ]
}
```

**④ 전역 + 일부 제외 (단 T003·T004는 수집 안 함):**

```json
{
  "scope": "GLOBAL_WITH_EXCLUDE",
  "global": { "override_type": "RANGE", "start_dt": "2026-07-01 00:00:00", "end_dt": "2026-07-05 00:00:00" },
  "exclude": ["T003", "T004"]
}
```

### 6-3. 매칭·적용 로직

우선순위: **테이블별(TARGET) > 전역(GLOBAL) > 평시 로직.** `target_id`로 매칭한다. `TARGET`·`GLOBAL_WITH_EXCLUDE`는 대상 행을 INSERT에서 걸러 해당 테이블이 수집되지 않게 한다.

```sql
-- j := '@{pipeline().parameters.P_OVERRIDE_JSON}'::jsonb

-- (1) 수집 대상 필터 (base WHERE)
AND CASE j ->> 'scope'
        WHEN 'TARGET' THEN
            t.target_id IN (SELECT elem ->> 'target_id'
                            FROM jsonb_array_elements(COALESCE(j -> 'targets','[]'::jsonb)) elem)
        WHEN 'GLOBAL_WITH_EXCLUDE' THEN
            t.target_id NOT IN (SELECT jsonb_array_elements_text(COALESCE(j -> 'exclude','[]'::jsonb)))
        ELSE TRUE
    END

-- (2) 유효 오버라이드(eff) 결정
LEFT JOIN LATERAL (
    SELECT elem FROM jsonb_array_elements(COALESCE(j -> 'targets','[]'::jsonb)) elem
    WHERE elem ->> 'target_id' = t.target_id LIMIT 1
) ov ON true

CASE
    WHEN ov.elem IS NOT NULL                      THEN ov.elem        -- 테이블별
    WHEN j ->> 'scope' = 'GLOBAL'                 THEN j              -- 전역(단일)
    WHEN j ->> 'scope' IN ('GLOBAL_WITH_EXCEPTION','GLOBAL_WITH_EXCLUDE')
                                                  THEN j -> 'global'  -- 전역(혼합/제외)
    ELSE NULL                                                         -- 평시 로직
END AS eff
```

`eff`가 NOT NULL이면 `start_dt`/`end_dt` 상수로 추출 쿼리를 만들고(A·B 모두 상수 경로), NULL이면 각 방안의 평시 로직을 탄다. `TARGET`·`GLOBAL_WITH_EXCLUDE`에서 걸러진 행은 `ctl_ingest_pipeline_run` 레코드가 생성되지 않아 **수집 자체가 수행되지 않는다.**

## 7. SQL Injection 방어 (주로 방안 A / Override 경로)

메타에서 값을 문자열로 결합하는 구간(방안 A의 추출 쿼리 조립, Override 상수 삽입)에 적용한다.

### 7-1. 역할 분리 — ADF는 값만, PG가 SQL 생성

ADF 표현식(`@{...}`)이 SQL 텍스트를 조립하지 않게 하고 **값만** 넘긴다. SQL 생성·이스케이프는 메타(PG)의 `format()`에서 처리한다. ADF에는 SQL 리터럴 이스케이프 함수가 없기 때문이다.

### 7-2. `format()` + `quote_literal(%L)`

값 결합 시 `'...' || val || '...'` 대신 `format()`의 `%L`(quote_literal)을 쓴다. 작은따옴표 자동 이스케이프로 injection과 따옴표 깨짐이 동시에 해결된다. 식별자(스키마/테이블/컬럼)는 `%s`로 원문 사용(신뢰된 메타 값).

```sql
-- 취약: 값에 ' 들어오면 깨지거나 주입
' where ' || col || ' >= ''' || p_start_dt || ''''
-- 안전: %L 자동 인용·이스케이프
format(' where %s >= %L', col, p_start_dt)
```

### 7-3. 타입 캐스팅으로 형식 강제

날짜 값은 결합/비교 전 형식 검증(캐스팅 또는 정규식). `'2026-07-01'; DROP TABLE...` 같은 값은 캐스팅에서 에러가 나 실행 전 차단된다.

### 7-4. `WHERE_CLAUSE` 통짜 전달 미채택 (확정)

SQL 절을 통째로 받는 방식은 미채택. Override는 `RANGE`(start_dt/end_dt)만 허용하며, 이 값들은 7-2·7-3의 이스케이프·검증을 거친다. 사용자가 SQL 구문 자체를 주입할 경로가 없다.

### 7-5. 심층 방어

- 입력 주체를 **내부 운영자/관제 시스템으로 한정**.
- 소스 DB 접속 계정을 **읽기 전용(SELECT only)**으로 부여.
- 생성된 `extract_query`를 실행 전 이력 테이블에 로깅 → 이상 쿼리 사후 탐지.

## 8. 주의·확인 필요 사항 (⚠)

**공통**

- ⚠ **상한선(`<`) 필수** — 두 방안 모두 하한선만 두지 말 것(3절). B 예시(`>= current date - 1 days`)에 상한선 추가 확인.
- ⚠ **`scope` enum 표준** — `GLOBAL`/`TARGET`/`GLOBAL_WITH_EXCEPTION`/`GLOBAL_WITH_EXCLUDE` 4종 확정. 운영 문서·검증 로직에 동일 명칭.
- ⚠ **수집 제외 = 레코드 미생성** — `TARGET`(미지정분)·`GLOBAL_WITH_EXCLUDE`(exclude분)는 레코드를 만들지 않는다. "제외 이력"이 필요하면 SKIP 상태 레코드 등 대안 검토.
- ⚠ **지연 도착 데이터** — `>= / <` 구간만으로는 늦게 커밋된 데이터가 누락될 수 있음. 필요 시 lookback 정책 별도 검토.

**방안 A (ADF 계산)**

- ⚠ **ADF TZ 변환** — ADF 기본 UTC. `convertTimeZone`/`convertFromUtc`로 대상(소스) TZ 변환 후 계산해야 함.
- ⚠ **DB2 캐스트 표기** — `TIMESTAMP('...')`/`DATE('...')`가 표준 문자열이면 동작하나, 애매하면 `TIMESTAMP_FORMAT`/`TO_DATE`가 안전. 실제 DB2 버전에서 테스트.
- ⚠ **`incr_column_type` 표준화** — 메타에 `TIMESTAMP`/`DATE`/`TIME` 등이 일관 저장돼 있는지 확인(변형 표기 시 CASE 매핑 추가).
- ⚠ **자정 경계** — ADF 트리거 시각 기준. 일 배치는 소스 TZ 자정에서 오프셋을 둔 시각(예: 00:05~00:30) 트리거 권장.

**방안 B (소스 함수)**

- ⚠ **dialect 유지보수** — DB2/Oracle 등 소스 유형별 함수식 관리 필요.
- ⚠ **인덱스 사용** — 인라인 함수(`TRUNC(...)`, `current date - N`)가 컬럼 인덱스를 못 탈 수 있음. 대용량 테이블 실행계획 확인.
- ⚠ **추적성 한계** — `extract_query`에 함수가 저장되어 나중에 실행하면 재계산됨. "당시 구간"을 남기려면 별도로 실행 시각·해석값을 기록하는 보완책 검토.
- ⚠ **실행 지연·자정** — 소스 실행 시점 기준이라, 배치 지연이 자정을 넘으면 구간이 달라질 수 있음.
